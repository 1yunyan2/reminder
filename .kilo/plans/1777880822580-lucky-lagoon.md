## 数据分析：引脚映射与实际铜箔不匹配

| 铜箔 | 代码认为接在 | 实际触摸时哪个通道高 | 实际接在 |
|------|-------------|-------------------|---------|
| 头部 | GPIO1 (h) | h=222244 ✓ | GPIO1 ✓ |
| **腹部** | GPIO7 (a) | **p=382万 (GPIO3)**，a=40 ✗ | **GPIO3** |
| **背部** | GPIO2 (b) | **a=20万 (GPIO7)**，b=28 ✗ | **GPIO7** |
| 前翻页 | GPIO3 (p) | p=190431 ✓ | GPIO3（与腹部冲突!） |
| 后翻页 | GPIO14 (n) | n=39413 ✓ | GPIO14 ✓ |

### 关键发现

1. **腹部铜箔实际接在 GPIO3**（代码定义为前翻页），不是 GPIO7
2. **背部铜箔实际接在 GPIO7**（代码定义为腹部），不是 GPIO2
3. **GPIO2（代码定义为背部）实际没有连接任何铜箔**，或接了其他东西
4. 后翻页触摸时 h 从 -208 暴增到 3,813,567 —— 这是**突变而非渐变耦合**，可能是触摸传感器读数在 200ms 延期间产生溢出/错误

### 后翻页触发情绪的真正原因

后翻页（GPIO14）触摸时，头通道（GPIO1）h 值从 -208 暴增到 **3,813,567**。这不是正常的电容耦合（正常耦合应该是几千的渐变），而是读数异常。

可能原因：
- GPIO14 和 GPIO1 的 PCB 走线非常近，产生串扰
- 200ms 的扫描间隔导致触摸传感器状态机异常（正常应该是 15-20ms）
- 触摸传感器读数溢出（ESP32-S3 的 touch_pad_read_raw_data 返回值范围问题）

## 修复方案

### 方案 A：修正引脚映射（解决腹部/背部问题）

修改 `bsp_config.h`：
```c
// 修正前：
#define BSP_TOUCH_2_PIN 2     // 背部
#define BSP_TOUCH_3_PIN 7     // 腹部

// 修正后：
#define BSP_TOUCH_2_PIN 7     // 背部 → 实际接在 GPIO7
#define BSP_TOUCH_3_PIN 3     // 腹部 → 实际接在 GPIO3
// 前翻页需要换到其他空闲引脚（GPIO2 空闲了）
#define BSP_TOUCH_PREV_PIN 2  // 前翻页 → GPIO2 空闲，但前翻页实际也接在 GPIO3?
```

**问题：** 腹部和前翻页都显示接在 GPIO3 上。这说明可能只有一个铜箔接在 GPIO3 上（腹部），而代码的 `btn_prev_page` 实际上读的是腹部铜箔的数据。

**需确认：** 实际硬件上 GPIO2、GPIO3、GPIO7 分别连接的是什么铜箔？

### 方案 B：解决后翻页 h 值暴增问题

1. **移除 200ms 诊断延迟**（恢复 15ms 正常扫描周期），重新测试 h 值是否还会暴增
2. 如果暴增仍然存在，说明是 PCB 串扰问题，需要用软件方案：
   - 当检测到 h 值突变（变化量 > 阈值）时，忽略该周期的 h 值
   - 或者：当 n > 阈值（后翻页被触摸）时，强制屏蔽 h 通道

### 方案 C：后翻页触摸时彻底屏蔽头部（软件兜底）

不管 h 值暴增的原因是什么，当后翻页通道有信号时，直接屏蔽头部：
```c
if (dn > thresh_abs)  // 后翻页有信号
    dh = 0;           // 清零头通道，从源头消除
```
